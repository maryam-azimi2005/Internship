**Client/Server**

در معماری وب، ارتباط بین اجزای مختلف معمولاً بر اساس مدل Client / Server انجام می‌شود. در این مدل، کلاینت به بخشی گفته می‌شود که ریکوئست ها را ارسال می‌کند و سرور بخشی است که آن ریکوئست ها را دریافت و پردازش می‌کند و پاسخ می‌دهد. مرورگرهای وب مانند (Chrome یا Firefox) و ابزارهایی مانند Postman نمونه‌هایی از کلاینت هستند، در حالی که بک‌اند یک وب‌سایت یا یک API که با Django پیاده‌سازی شده، نقش سرور را دارد. سرور معمولاً روی یک آدرس و پورت مشخص اجرا می‌شود و به‌ صورت مداوم در حال listen کردن است تا در صورت دریافت ریکوئستی از سمت کلاینت‌ ها، آن را پردازش کند.

وقتی یک کلاینت می‌خواهد به اطلاعاتی دسترسی پیدا کند، یک ریکوئست به سرور ارسال می‌کند. این ریکوئست شامل اطلاعاتی مانند آدرس (URL) ، نوع درخواست (مثلاً GET یا POST) و گاهی موارد دیگر است. سرور که در حالت انتظار(listen) قرار دارد، پس از دریافت ریکوئست آن را بررسی می‌کند و منطقی که برای آن نوشته ایم را اجرار میکند (برای مثال خواندن داده از پایگاه داده یا اعمال قوانین دسترسی) و در نهایت یک پاسخ (Response) به کلاینت برمی‌گرداند. این پاسخ معمولاً شامل یک Status Code برای مشخص کردن موفق یا ناموفق بودن درخواست و داده‌هایی به شکل HTML یا JSON است.

به عنوان نمونه، زمانی که کاربر یک URL را در مرورگر وارد می‌کند، مرورگر در نقش کلاینت یک ریکوئست از نوع GET برای سرور ارسال می‌کند. سرور که از قبل روی این آدرس فعال بوده و منتظر دریافت ریکوئستهاست، ریکوئست را دریافت می‌کند، مسیر مربوطه را بررسی میکند و در صورت ولید بودن، اطلاعات مورد نظر را آماده کرده و در قالب یک Response به مرورگر بازمی‌گرداند. در نهایت، مرورگر این پاسخ را دریافت کرده و محتوای آن را به شکلی مناسب برای کاربر نمایش می‌دهد.

**HTTP Methods**

در ارتباطات وب، برای مشخص کردن نوع عملیاتی که کلاینت قصد انجام آن را دارد، از HTTP Methods استفاده می‌شود. هر ریکوئست که از سمت کلاینت به سرور ارسال می‌شود، حتماً دارای یک HTTP Method است که به سرور می‌گوید این درخواست با چه هدفی ارسال شده است؛ مثلاً دریافت داده، ارسال داده یا حذف آن. رایج‌ترین HTTP Methods شامل GET، POST، PUT و DELETE هستند. متد GET برای دریافت اطلاعات از سرور استفاده می‌شود و معمولا تغییری در داده‌ های سرور ایجاد نمی‌کند. متد POST برای ارسال دیتای جدید به سرور به‌کار می‌رود، مانند ثبت‌نام یک کاربر جدید. متد PUT برای به‌روزرسانی دیتای موجود و متد DELETE برای حذف دیتا استفاده می‌شود.

**Status Code**

در ارتباطات وب، هر Response که از سمت سرور به کلاینت ارسال می‌شود، شامل یک Status Code سه‌رقمی است که نتیجه‌ی پردازش ریکوئست را مشخص می‌کند. این کد ها به دسته‌های مختلفی مانند 2xx برای ریکوئست های موفق، 4xx برای خطاهای مربوط به کلاینت (مانند 404) و 5xx برای خطاهای سمت سرور (مانند 500) تقسیم می‌شوند. سرور پس از پردازش ریکوئست، Status Code مناسب را در Response قرار می‌دهد وکلاینت با بررسی آن می‌تواند تشخیص دهد که درخواست با موفقیت انجام شده، نیاز به اصلاح دارد یا با خطا مواجه شده است.

**REST**

در معماری REST، نحوه‌ی طراحی و برقراری ارتباط بین کلاینت و سرور بر اساس مجموعه‌ای از اصول مشخص انجام می‌شود. در این معماری، سرور منابع مختلف را به‌صورت resource در اختیار کلاینت میگذارد و هر resource از طریق یک URL مشخص قابل دسترسی است. کلاینت با استفاده از HTTP Methods مانند GET، POST، PUT و DELETE روی این resourceها عملیات مختلف انجام می‌دهد و سرور نیز بدون نگه‌داشتن وضعیت کلاینت (stateless) ریکوئست ها را پردازش کرده و Response مناسب را معمولا به شکل JSON همراه با Status Code به کلاینت برمیگرداند.

**JSON api**

همانطور که گفته شد در معماری REST، داده‌هایی که بین کلاینت و سرور رد و بدل می‌شوند معمولاً به شکل JSON api هستند. در این حالت، سرور پس از پردازش ریکوئست، دیتا را در قالب JSON به کلاینت ارسال می‌کند تا به‌ راحتی قابل پردازش توسط برنامه ‌ها باشد. برای مثال، اگر کلاینت یک ریکوئست از نوع GET به آدرس /users ارسال کند، سرور ممکن است یک Response با Status Code 200 و داده‌ای به شکل زیر برگرداند:

{ "id": 1, "name": "Ali" }

در این مثال، استفاده از JSON api باعث می‌شود کلاینت بدون وابستگی به نمایش ظاهری، فقط دیتای مورد نیاز را دریافت و استفاده کند.

**Postman app**

یک ابزار برای تست و بررسی API ها در ارتباطات وب است که نقش یک کلاینت را بازی می‌کند. با Postman می‌توان ریکوئست های مختلف را با HTTP Methods مثل GET یا POST به یک سرور ارسال کرد، آدرس URL و در صورت نیاز دیتای ریکوئست را مشخص کرد، و سپس Response را بررسی کرد. خروجی سرور معمولا شامل Status Code و بدنه‌ی پاسخ است و Postman کمک می‌کند بدون نیاز به ساختن یک رابط کاربری یا نوشتن کد اضافی، رفتار API را سریع و دقیق تست کنیم و مطمئن شویم طبق انتظار کار می‌کند.
